Landing_page.html review vs static mockup

From the rendered HTML snapshot:

- Structure:
  - The dynamic landing page correctly follows the designed sections:
    - Hero with left text and right cards.
    - “Why this system works” section.
    - “How it works (for patients)” steps.
    - Seniors & one-clinic section.
    - CTA band.
    - Footer.
- Tokens and components:
  - ui-section, ui-card, ui-badge, ui-btn classes are present.
  - Mantine components render correctly with semantic structure.

However, issues that make it look “weird and ugly” versus the static mockup:

1) Visual hierarchy:
   - The hero lacks the strong contained width, vertical spacing, and breathing room of the static mockup.
   - Typography (sizes/weights) is still Mantine defaults; not tuned to the static’s polished scale.
   - CTAs and cards visually compete rather than forming a clear focal point.

2) Button semantics and appearance:
   - UiButton currently wraps Mantine Button directly without proper link semantics:
     - Using Button with href in HTML output (type="button" href="...") is incorrect for navigation and may not visually match anchor-style CTAs.
   - The large Mantine class stack plus ui-btn classes can look noisy and inconsistent with the simple, pill-shaped buttons from static/styles/globals.css.

3) Card and section spacing:
   - ui-card & ui-section tokens are basic; paddings/margins aren’t fully tuned to match the static layout.
   - There’s no top-level max-width / centered layout resembling the designed breakpoint behavior.
   - Some mini cards (e.g., staff portal) appear cramped.

4) Static → dynamic fidelity:
   - The static mockup has:
     - A tighter hero/CTA composition.
     - Specific visual rhythms (card shadows, spacing, subtle borders).
     - Strong contrast and clear vertical rhythm.
   - The current dynamic implementation is “correct” structurally but still looks like Mantine defaults plus light token overrides.
